### Problem Status Transitions

By default, a problem is constrained by the different states in its lifecycle. For example, when a problem is in the logged state, the next state is investigation. From investigation, it can be set to identified. (See :Link[Problem State Transitions Reference]{label="Problem State Transitions Reference" overridedLabel="Problem State Transitions Reference" spaceId docId="3UII0SQsYZe7S_VW1xyi6" version="v2" docAnchorId="#Problem" loadingMethod="dynamic" newTab="false" githubPath="Problem_State_Transitions.htm#Problem" href="Problem_State_Transitions.mdx"} for a list of the problem states.)

### Adding a Custom State to the List of Problem States

Follow these steps to add a custom state to the list of problem states that is available in \{\{FRS\_VARS.ProductName}}:

1. From the problem record, click **Edit** in the **Status** field. The **Status** window appears.
2. Click **Add**. The **New Problem Status Workflow** window appears.

<Paragraph indent="1">
  *New Problem Status Workflow*
</Paragraph>

<Image src="https://archbee-image-uploads-qa.s3.amazonaws.com/yzXkkCTORWlcwRG_AVHKZ/gcXAiXYiuPUF1Zkglbk7t_newproblemsatusworkflow.png" position="flex-start" size="40" indent="1" initialPath="../../Resources/Images/help-images/new_problem_satus_workflow.png" githubPath="Content/Resources/Images/help-images/new_problem_satus_workflow.png" showCaption="false"></Image>

3. Enter information into the fields.

| Field          | Description                                           |
| -------------- | ----------------------------------------------------- |
| Current Status | The current status. Probably logged.                  |
| Next Status    | The name of the new status.                           |
| Sort Order     | The number of the order the current status should be. |

4. Click **Save**.
5. Choose the new status to add it to your problem.
6. Click **Save**. All new problems now have the new status available from the drop-down list.
