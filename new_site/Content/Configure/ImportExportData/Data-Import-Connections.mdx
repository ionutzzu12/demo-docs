### Comma Delimited (CSV)

<Paragraph indent="1">
  Most applications, including Microsoft Word and Microsoft Excel, can read CSV files. However, this format does not support line breaks. Do not include including commas in the data of the last column. If a value in the last column contains a comma even when enclosed in quotes CSV interprets it as a field separator.
</Paragraph>

<Paragraph indent="1">
  For example: "Initials", "mail", "telephoneNumber", "givenName", "company", "Wilma", "wasselt", "Asselt", "van".
</Paragraph>

<Paragraph indent="1">
  In this example, the last field represents lastName, with the value "Asselt, "van". If this field appears in any column other than the last, the import correctly recognizes "Asselt, "van" as a single value. However, if it is in the last column, the import reads only "Asselt" and ignores the data after the comma.
</Paragraph>

### Excel Spreadsheet (XLS, XLSX)

Microsoft Excel and other compatible applications can read these file formats. You can import only a single worksheet per Excel file. You can import up to 5,000 rows per file. If your file contains more than 5,000 records, split it into smaller files and import each file separately.

### About Multiple Role Support for CSV and XLS files

To support multiple roles, add a column named **Role** (including the asterisks) to the input file. This column must contain the role display names, not the internal role names.

- Use display names like **Self Service**, not internal names like **SelfService**.
- Separate multiple roles in the same cell with commas.

The application does not create new roles during import. You must define roles in the system before assigning them to employees.

When importing, the application merges roles by default. For example, if an employee currently has two roles Service Desk and IT and you import three roles IT, admin, and service owner the final result is four roles:

Service Desk, IT, admin, service owner.

### XML

Imports generic XML data.

| Field              | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |

In this format, the table element is "Rows", and row element is "Row".<br />\<Rows>&#xA;    \<Row>&#xA;        \<Field>PrimaryEmail\</Field>&#xA;        \<Field>FirstName\</Field>&#xA;         ...&#xA;    \<Row>&#xA;\<Rows><br />Do not check this option to use the flexible two-level format.<br />In this example,the table element is "Employee\_Info", and the row element is "Employee". Data with null values do not appear in the XML file, which can cause problems when exporting data.

For example, consider the following XML. Note the data for the second employee contains an email cell with no attributes. This cause the data to import incorrectly.

\<Employee\_Info>
    \<Employee Employee\_Number="105">
        \<Name>Masashi Okamura\</Name>
        \<Department>Design Department\</Department>
         \<Email>masashi\@design.com\</Email>
        \<Phone>408.555.1234\</Phone>
    \</Employee>
    \<Employee Employee\_Number="106">
        \<Name>Frank Smith\</Name>
        \<Department>Design Department\</Department>
         \<Email>\</Email>
        \<Phone>408.555.1235\</Phone>
    \</Employee>
\</Employee\_Info>

To alleviate this issue, create an empty string:

\<Employee\_Info>
    \<Employee Employee\_Number="105">
        \<Name>Masashi Okamura\</Name>
        \<Department>Design Department\</Department>
         \<Email>masashi\@design.com\</Email>
        \<Phone>408.555.1234\</Phone>
    \</Employee>

        \<Name>Frank Smith\</Name>

\</Employee\_Info>

To prevent errors while importing files containing currency data, map the currency value and the currency type separately.

### Reviewing and Publishing Import or Web Service Connection Data

*Data Import Setup Sequence -- Review and Publish*

<Image src="https://archbee-image-uploads-qa.s3.amazonaws.com/yzXkkCTORWlcwRG_AVHKZ/XbigtpaRaeN65u67BP6M7_reviewpublish.png" alt="Data import setup sequence - review and publish." position="flex-start" size="40" initialPath="../../Resources/Images/help-images/review_publish.png" githubPath="Content/Resources/Images/help-images/review_publish.png" showCaption="false"></Image>

Click the **Test Transformation** button in order to view all the mappings specified by you. The mapping configurations appear in a pop-up window as an xml file. If desired, you can copy and paste the information from this window to another file.

You have four options to review or publish your import data:

- **Preview**: Views the form with your field mappings. All records from the first batch appear showing the mapping configuration and the associated data. You can check these records for import errors before beginning import.
- **Publish**: The application only shows this option if there is a schedule involved, like with the **Secure FTP (SSH)** connection types. Activates the connection so that it can run according to the schedule.
- **Publish & Run Now**: Runs the import immediately and places it on the import schedule.
- **Publish & Test Run**: Runs the import with only the first batch of records. A batch is 200 records. If errors or problems occur in this job, you can go back to your data and troubleshoot before running the entire job.

The application automatically saves all field mappings even if they are not immediately published. After you publish the data, the job runs based on the schedule/s you selected.

### Example: Importing Configuration Item Relationships

This example shows how to import data for a configuration item (CI) map tree.

Previously, to define a relationship between a configuration item and an application, you used the Map Tree tab to link related configuration items based on a specific named relationship. This method still works, but it can be time‑consuming if you need to map many configuration items.

In \{\{FRS\_VARS.ProductName}} Release 2015.1,a new feature lets you use data integration to link multiple configuration items from a CSV or Microsoft Excel file. To use this feature, you must have:

- The DataIDs, SourceIDs, and TargetIDs of the configuration item records to link as well as
- The RecID of the **CINamedRelConstraints** record that defines the name of the relationship between the configuration items (such as "managed by" or "runs on").

This feature also requires the **CINamedRelLink** business object, which is new in \{\{FRS\_VARS.ProductName}} Release 2015.1. If you do not have that business object, contact \{\{FRS\_VARS.Company-short}}.
