\{\{FRS\_VARS.ProductName}} contains a user named **InternalServices** by default to support the workflow and escalation engines. This user is linked to the admin role. You cannot delete this user. Do not link it to other roles or unlink it from the admin role. Doing so will disable the workflow and escalation features.

An employee is an internal customer who uses \{\{FRS\_VARS.ProductName}}. The application provides employees with an account that defines the login access and available roles. The application assigns at least one role to all employees and they use this role to log into the application.

Employee accounts include general information (login and role information) and extended information (personal information).

<hint type="info">
  To add a large number of users within Active Directory, use the LDAP Active Directory Import tool. Refer to [Configuring LDAP Settings](../ImportExportData/LDAP_Settings.mdx).
</hint>

## Accessing the Employee List

1. Do one of the following:
   - From the Configuration Console, click **Configure > Users and Permissions > Users** to open the **Users** workspace.
   - Log in to the application and open the **Employee** workspace.
2. Open an employee record to view its properties.

## Creating an Employee

Employee information is stored in the **StandardEmployee** business object. Edit this business object to track information to capture about each employee in your application.

1. From the Configuration Console, click **Configure > Users and Permissions > Users** to open the **Users** workspace.
2. Click **New Employee** from the toolbar. The application displays an empty employee form. The following default tabs appear (results may vary depending on your application setup):
   - **Details**: Continue to the next step.
   - **Roles**: The roles to which the employee belongs. Refer to [Working with Roles](../../ServiceDesk/Roles/roles.mdx).
   - **Incident**: Incidents in which the employee is the customer. Refer to :Link[About Creating an Incident for an Employee]{label="About Creating an Incident for an Employee" overridedLabel="About Creating an Incident for an Employee" spaceId docId="gO65DKWOS0RTWm_hpdz38" version="v2" docAnchorId="#Create_an_Incident_for_Employees" loadingMethod="dynamic" newTab="false" href="Working_with_employees.mdx"}.
   - **Change**: Changes in which the employee is the requester. Refer to [Creating a Change Request](../../ServiceDesk/Change/Creating-a-Change-Request.mdx).
   - **Contact Group**: Refer to :Link[Adding an Employee to a Contact Group]{label="Adding an Employee to a Contact Group" overridedLabel="Adding an Employee to a Contact Group" spaceId docId="gO65DKWOS0RTWm_hpdz38" version="v2" docAnchorId="#Add_an_Employee_to_a_Contact_Group" loadingMethod="dynamic" newTab="false" href="Working_with_employees.mdx"}.
   - **CI**: Configuration items in which the employee is the owner. Refer to [About Creating a Configuration Item](../../ServiceDesk/ConfigurationMgmt/About_Creating_a_Configuration_Item.mdx).
   - **Activity History**: Refer to [About Activity History](../../ServiceDesk/ActivityHistory/Activity_History.mdx).
   - **Attachment**: Refer to [Working With Attachments and URLs](../Attachments/Attachments.mdx).
   - **Team**: The teams to which the employee belongs. Refer to [Working with Teams](Teams.mdx).
   - **Recurring Service Subscription**: The recurring service subscription is related to cost allocation as part of IT Finance Management (ITFM) and is used to allocate costs for any recurring services that an employee subscribes to.
   - **Time Off** - Request time off. Click **New Employee Time Off**, enter the required details and save. A time off is created and ready for approval.
   - **Accessible Org Units** - Employees can be added to accessible org units to control the and set access permissions as to who can access certain assets. If an employee is added to Brand Marketing and Communications Accessible Org Units, only assets that are linked to Brand Marketing and Communications org units are accessible by that employee. For more information on linking assets to Org Units, refer to Viewing and Associating Configuration Items with Organizational Units.

<hint type="info" indent="2">

</hint>

<Paragraph indent="2">
  e. Save and exit.
</Paragraph>

## Deleting Employees

Employees can not be deleted from the application. To remove an employee from the active list, set their status to terminated.

1. From the Configuration Console, click **Configure > Users and Permissions > Users** to open the **Users** workspace.
2. Open an employee record.
3. From the **Status** field, select **Terminated** from the drop-down list.
4. Click **Save**.

### Linking Roles to an Employee

An employee must be linked to a role before they can access \{\{FRS\_VARS.ProductName}}. You can link multiple roles to an employee record. Upon logging in, the employee can select a role from the list defined in this workspace.

1. From the Configuration Console, click **Configure > Users and Permissions > Users** to open the **Users** workspace.
2. Open the employee record.
3. Click the **Roles** tab.
4. From the toolbar, click **Link**. The application displays the **Link Roles** dialog box.
5. Choose a role from the list, then click **Select**. The role appears on the list.
6. Click **Save**. The role is added to the list for the user. Repeat to assign additional roles.

## Inserting Non-LDAP Authentication Providers to Employee Profiles

To provide non-LDAP authentication providers to employee profiles, you can configure a workflow triggered whenever an employee record is created or modified. This workflow simply executes a quick action. The quick action adds a child record to the **FRS\_ExternalLoginAssocProfileEmployee** business object with **AuthProvider** and **Login** fields. Get the RecID from the authentication provider table to add this child record to the **Employee** business object.

## Creating a Call-Back Method

You can create a field in which users can select their preferred method of call-back contact.

1. Create a validation business object containing the values that the customer can choose from. Refer to [Using Validation Business Objects](../PickLists/Validation_Objects.mdx) for a description on how to do this.
2. From the **Employee** business object, create a new text field called **CallbackMethod**.
3. Check **Validated** under **Pick List** and select the validation business object that you created above. Refer to [Working with Fields](../BusinessObjects/Using-Fields.mdx) for a description on how to do this.

## Creating an Employment Type

Use the **Employment Types** workspace to define various types of employees.

The default types of employees are:

- Permanent
- Contractor
- Contractor - Part Time

1. From the Configuration Console, click **Configure > Organization Profile > Employment Types** to open the **Employment Types** workspace.
2. Click **New Employment Type**. The application displays a blank field.
3. Enter a unique name for the employment type.
4. Click **Save**. The employment type now appears on the list.
