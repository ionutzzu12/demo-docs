This topic lists out the \{\{FRS\_VARS.Company}} \{\{FRS\_VARS.ProductName}} and \{\{FRS\_VARS.Company}} Neurons integration steps, some common issues faced during the integration, and troubleshooting details.

## Import Discovered Data from Ivanti Neurons

You can take advantage of the Ivanti Neurons device discovery capabilities in \{\{FRS\_VARS.ITSM}} by importing a package from Ivanti Marketplace. This enables you to restore existing devices in the Ivanti \{\{FRS\_VARS.ProductName}} CI/Hardware Assets workspace from Ivanti Neurons. Also, newly discovered Ivanti Neurons devices get synced to the \{\{Primary.Company}} \{\{FRS\_VARS.ProductName}} CI > Hardware Assets workspace.

<hint type="info">
  Applicable only for cloud customers. Device data can be imported from Ivanti Neurons only during the synchronization process.
</hint>

<ExpandableHeading>
  ## Package Selection

  **For Ivanti Neurons Device Sync**

  - New and existing \{\{FRS\_VARS.ProductName}} customers can apply the CMDB package (that is less than 1 MB) from Ivanti Marketplace for enabling the Ivanti Neurons device sync.
  - New and existing customers who are using \{\{FRS\_VARS.ProductName}} version 2020.3 or later can apply the Ivanti Neurons Integration package from the Ivanti Marketplace for enabling the Ivanti Neurons device sync.

  **For Complete Asset Management Capabilities**

  New and existing \{\{FRS\_VARS.ProductName}} customers can apply the AMC (Asset Management Cloud) package from Ivanti Marketplace.

  ### Ivanti Neurons to \{\{FRS\_VARS.ProductName}} Integration - Device Sync Only

  1. Download the package (CMDB or the AMC) from Ivanti Marketplace.
  2. Validate and apply the package if there are no errors on the \{\{FRS\_VARS.ProductName}} 2020.3 upgraded tenant.
</ExpandableHeading>

  ## Enable the Decompress Message Quick Action

  By default, the Decompress Message Quick Action will be disabled for a device message. To enable the Decompress Message Quick Action of the Message Queue Journal business object, follow these steps:

  1. Open the Configuration console > **Build** > **Automation Tools** > **Quick Actions**.
  2. Click **Frs\_MessageQueueJournal** from the **Action Manager** list.
  3. Click the **Quick Action** tab.
  4. On the left side panel, click **Decompress Message**.
  5. Click **Edit** to open the **Edit Expression** window.
  6. Add Device type in the **Quick Action** expression field **Disable in UI** as shown below.

  <ExpandableHeading>
    ## Device Filter in Mapping

    The Filter Manager in **IN connector** (Ivanti Neurons Connector- Device) enables filtering inflow of device types based on the filter criteria. For instance, you can select to import **Asset Type** only of **Laptop** by creating the filter value as **Laptop**.

    *Filter Manager*

    <Image src="https://archbee-image-uploads-qa.s3.amazonaws.com/yzXkkCTORWlcwRG_AVHKZ/yG80dwu8hI7VbaSWfHk2a_filtermanager.png" alt="Use the Filter Manager to manage filter criteria." position="flex-start" size="40" initialPath="../../Resources/Images/Configure/ISM-Neurons-Integration-BestPractics/FilterManager.png" githubPath="Content/Resources/Images/Configure/ISM-Neurons-Integration-BestPractics/FilterManager.png" showCaption="false"></Image>



    <hint type="info">
      - The filter settings made during import remain saved for further import.
      - Based on the filter settings, only those messages which are relevant to the mapping will be processed by the \{\{Primary.Company}} \{\{FRS\_VARS.ProductName}} tenant and rest of the messages will be discarded.
      - To filter out null parameters, use the NotEqual operator instead of NoContain. For example, when filtering a device that does not have serial number to sync from Neurons.
      - The filter settings work only for a single value filter in each row. If the field selected in the left panel of the mapping results in multiple values, then those filters will not work. You are advised to use the filter settings accordingly.
    </hint>



    For example, if you choose Device.BIOS.BIOSSettings in the root table, all the properties related to Device.BIOS.BIOSSettings will be displayed on the **Filter settings** page.

    <Image src="https://archbee-image-uploads-qa.s3.amazonaws.com/yzXkkCTORWlcwRG_AVHKZ/i0bw2G-OPMSE4Pun5qwvz_biostable.png" alt="The settings shown in the Root Table." position="flex-start" size="40" initialPath="../../Resources/Images/Configure/ISM-Neurons-Integration-BestPractics/BiosTable.png" githubPath="Content/Resources/Images/Configure/ISM-Neurons-Integration-BestPractics/BiosTable.png" showCaption="false"></Image>

    ## Export and Import the Ivanti Neurons Connector

    You can now edit the **IN** Connector mapping and export it as a metadata package and import on other \{\{Primary.Company}} \{\{FRS\_VARS.ProductName}} tenants on demand. The exported connector will include the default Table selection, Filter option, and Field Mapping changes.

    This functionality is intended to help the Asset Administrator have a controlled import in STG/UAT, verify the imported asset, and update the mapping before importing all the assets in a PRD landscape.



    <hint type="info">
      - This feature is applicable only for an **Update** scenario; a package exported from here will not create a new connection if the connector does not exist.
      - Post import of the package, you are advised to review the changes once before you start syncing the devices.
      - Clone and delete actions for Ivanti Neurons Connector – Device are disabled or removed.
    </hint>

<ExpandableHeading>
  ## Why is the Neurons data to Neurons for ITSM integration device sync slow?

  <Paragraph indent="1">
    The device sync depends on few factors such as:
  </Paragraph>

  <Paragraph listStyleType="disc" indent="2">
    Inflow of device messages from Ivanti Neurons to \{\{FRS\_VARS.ProductName}}.
  </Paragraph>

  <Paragraph listStyleType="disc" indent="2">
    Rate at which \{\{FRS\_VARS.ProductName}} can read the device messages from Ivanti Neurons. Currently we are facing some technical challenges in terms of device processing rate and creating/updating asset in real time.
  </Paragraph>

  <Paragraph indent="1">
    The solution is to raise a service ticket to Support, the support team will analyze and take necessary actions.
  </Paragraph>
</ExpandableHeading>

<ExpandableHeading>
  ## How to track the delay in sync in the Neurons data to Neurons for ITSM

  <Paragraph indent="1">
    The **MessageQueueJournal** workspace will display the latest received device message along with software messages. One can make out the device processed vs the device processing failed. The dashboard will be added to have details on how many device messages received, how many failed etc.
  </Paragraph>
</ExpandableHeading>

<ExpandableHeading>
  ## Is there an ability to force a sync or option to sync specific device?

  <Paragraph indent="1">
    This feature is not available. The messages are read in FIFO order and hence the more recent scan device message takes longer time to sync in \{\{FRS\_VARS.ProductName}}.
  </Paragraph>
</ExpandableHeading>

## Troubleshooting

<ExpandableHeading>
  ## When performing data sync from Neurons Discovery to Neurons for ITSM, system performance is degraded

  **Resolution:**

  When you schedule the data sync, ensure that the max size is no greater than 500. For more information, refer to [Scheduling API Data Sync from Ivanti Neurons for Discovery to Ivanti Neurons for ITSM](<../ImportExportData/Scheduling Sync From Neurons Discovery to ISM.mdx>).
</ExpandableHeading>

<ExpandableHeading>
  ## When scheduling data sync from Neurons Discovery to Neurons for ITSM, Preview fails.

  **Resolution:**

  Verify that the download is no greater than 20 MB. Sync packages that are larger than 20 MB will cause the preview to fail. For more information, refer to [Scheduling API Data Sync from Ivanti Neurons for Discovery to Ivanti Neurons for ITSM](<../ImportExportData/Scheduling Sync From Neurons Discovery to ISM.mdx>).
</ExpandableHeading>

<ExpandableHeading>
  ## Opening Unmanaged Throws Error

  **Resolution:**

  1. Create CI extended BO ivnt\_UnmanagedAsset
  2. Add forms by AM.UnmanagedAsset.Brief
  3. Open and select Layout of CI (relevant) > Open in Grid view > Add the ivnt\_UnmanagedAsset with newly created form.
  4. Open Layout of the CI BO and select Layout of CI (relevant) > Open in Form view > Add the ivnt\_UnmanagedAsset with newly created form.
</ExpandableHeading>

<ExpandableHeading>
  ## Object Reference NULL Error

  **Resolution:**

  Can be observed by multiple reasons and some of the primary reasons could be:

  1. Check Ivanti Neurons Connector - Device Test Connection.
  2. Missing OData Connection information like API path Object.

  <Image src="https://archbee-image-uploads-qa.s3.amazonaws.com/yzXkkCTORWlcwRG_AVHKZ/Y1SH5W773BVNzfx5Q_VC1_configdetails.png" position="flex-start" size="40" indent="1" initialPath="../../Resources/Images/Configure/ISM-Neurons-Integration-BestPractics/ConfigDetails.png" githubPath="Content/Resources/Images/Configure/ISM-Neurons-Integration-BestPractics/ConfigDetails.png" showCaption="false"></Image>

  3. More than one OData Connections defined: This can happen if customer has already created an OData Connector ford Device import manually before importing the \{\{FRS\_VARS.ProductName}} CMDB Turnkey package form Marketplace.

  <hint type="info">
    At a time only one OData Connector can be active for connection name Device.
  </hint>

  4. Ivanti Neurons Connector-Device type is disabled in Connector page. It’s advised to create the default naming as Device if your using multiple mappings.
</ExpandableHeading>

<ExpandableHeading>
  ## Device Creation Failure

  **Resolution:**

  1. Enable the **Asset Processor Configuration Missing Create CI** checkbox.
  2. Mandatory fields for creating CI missing, eg: Hostname, CIType etc.
  3. Duplicate of Serial Number, Asset Tag, CI Component etc.
  4. Connector Test Connection failure.
</ExpandableHeading>

<ExpandableHeading>
  ## Device Type Not Updating

  **Resolution:**

  1. If the Ivanti Neurons does not have Device Type such devices will honor the Asset Processor Configuration.
  2. If Connector doesn't have the mapping details for Device Type which present in Neurons such devices will honor the Asset Processor Configuration.
</ExpandableHeading>

<ExpandableHeading>
  ## Device Field Values Are Not Populated

  **Resolution:**

  Review the mapping details in the connector.
</ExpandableHeading>

<ExpandableHeading>
  ## How to initiate the device scan on Neurons agent manually?

  **Resolution:**

  Run the following command in selected devices' console in the Neurons page:

  Start-Process -WorkingDirectory "C:\Program Files\Ivanti\Ivanti Cloud Agent\UNO.INVENTORY.ENGINE64" -FilePath "C:\Program Files\Ivanti\Ivanti Cloud Agent\UNO.INVENTORY.ENGINE64\Inventory.Engine.exe"

  The below command can list scanned items on the console itself:

  Get-ChildItem -Path "C:\Program Files\Ivanti\Ivanti Cloud Agent\UNO.INVENTORY.ENGINE64"
</ExpandableHeading>

<ExpandableHeading>
  ## Issue where the Neurons connector doesn not kick off the schedule nor does the Run Now option work

  **Resolution:**

  Possible reason could be the same agent has been installed on multiple Neurons tenants. Uninstall one of the agents, and other should start working.
</ExpandableHeading>
